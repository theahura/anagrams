# Noridoc: src

Path: @/src

### Overview

- Vue 3 SPA implementing Anagrams (Snatch-style word game): pure-function domain core under `src/` with Vue components under [`src/components/`](@/src/components).
- Domain modules are framework-agnostic, side-effect-free, and consumed by the orchestrator [`src/components/App.vue`](@/src/components/App.vue).
- Two persistence-relevant boundaries exist: dictionary asset loading (fetch of `data/dictionary.json` and `data/lemmas.json`) and Daily Challenge stats persistence via [`src/storage.js`](@/src/storage.js) → `localStorage`.

### How it fits into the larger codebase

- The application has no backend. All state lives either in Vue refs (transient) or in `localStorage` (Daily Challenge history only). There is no server, no API client, and no shared cache module.
- [`src/components/App.vue`](@/src/components/App.vue) is the sole orchestrator: it owns screen routing (`home` / `game` / `score`), constructs the game via `createGame`, advances it via `GameScreen`, and is the only caller of [`src/storage.js`](@/src/storage.js). Domain modules never import storage or Vue.
- Daily-mode determinism flows: a `YYYY-MM-DD` local-calendar date string (produced by [`src/dateKey.js`](@/src/dateKey.js)) seeds [`src/prng.js`](@/src/prng.js) inside `createGame` ([`src/game.js`](@/src/game.js)), so every player on the same local-calendar day gets the same shuffle. The same date string is the storage record key in [`src/storage.js`](@/src/storage.js). [`src/dateKey.js`](@/src/dateKey.js) is the single source of truth for that string (`formatLocalDate` / `todayLocal`).
- Share-text generation ([`src/share.js`](@/src/share.js)) is purely derived from the end-of-game `result` plus the current `streak` value computed from storage. It is the only public-facing serialization of game state.
- **Maintainer-time data pipeline.** Two independent offline scripts under [`scripts/`](@/scripts) generate the static assets in `data/`: `build-dict.js` produces `data/dictionary.json` (the TWL06 wordlist), and `build-lemmas.js` produces `data/lemmas.json` (a precomputed inflection→lemma lookup). The lemma script imports `wink-lemmatizer` (the only place it appears in the project) and is therefore a `devDependencies`-only entry. `npm run build-dict` and `npm run build-lemmas` are independent — regenerating one does not regenerate the other. Both JSON outputs are committed and shipped as static assets.

### Core Implementation

- **Game lifecycle.** [`src/game.js`](@/src/game.js) exposes `createGame`, `drawTile`, `submitWord`, `endGame`. State is immutable: each operation returns a new game object. `INITIAL_REVEAL = 3` tiles are drawn at construction. `endGame` computes `finalScore`, the longest word, and a parent-chain reconstruction (`buildChain`) used by the Score screen.
- **Pool model — three pools.** [`src/pool.js`](@/src/pool.js) defines the canonical board shape: `{ faceDown, looseLetters, words }`. `faceDown` is the unrevealed bag; `looseLetters` are revealed but uncommitted letters; `words` are committed plays, each carrying its `parents` (the prior words consumed to form it). `formWord` removes consumed loose letters via multiset subtraction and replaces consumed parent words with the new word.
- **Word legality.** [`src/anagramRules.js`](@/src/anagramRules.js) enumerates subsets of existing `words` (bitmask up to 16) and checks whether the typed word can be assembled from a chosen subset plus loose letters. Failures are prioritized (`letters-unavailable` < `no-new-play` < `trivial-inflection`) so the worst applicable reason is reported. The `no-new-play` reason fires for any consume-without-growth play — when a non-empty parent subset is chosen and `typed.length <= max(parent.length)`. This covers both exact replay (single parent, typed === parent) and same-length single-parent rearrangement (e.g. `side` → `dies`); multi-parent merges always pass because the typed word's length necessarily exceeds the longest parent. Profanity is filtered via `bad-words`. Trivial inflections (e.g. plural-by-S) are rejected via [`src/trivialInflection.js`](@/src/trivialInflection.js); `canFormWord` threads `dict.lemmaIndex` through to that check.
- **Trivial-inflection lookup is precomputed.** [`src/trivialInflection.js`](@/src/trivialInflection.js) is a pure function `isTrivialInflection(answer, root, lemmaIndex)`: it consults `lemmaIndex` (accepting either a `Map<string, string[]>` or a plain object) for `answer`'s lemmas and returns `true` when `root` is one of them. A trailing-`e` heuristic also fires when `answer.startsWith(root)` (or matches `dropE + 'ing'`) and the dropped-`e` form appears in the lemma list. The runtime no longer imports any lemmatization library — all `wink-lemmatizer` work happens at maintainer time. The two override sets `FORCE_TRIVIAL` and `FORCE_NON_TRIVIAL` (keyed `root|answer`) live alongside the runtime function so override edits don't require regenerating `data/lemmas.json`.
- **Scoring.** [`src/scoring.js`](@/src/scoring.js) scores each word as `length²` and subtracts `MISSED_DRAW_PENALTY (10)` per missed-draw. `hasLoosePoolAnagram` (used by `drawTile` to detect missed draws) enumerates loose-letter subsets up to size 8 and checks them against the dictionary's `signatureIndex`.
- **Staging / highlight model.** [`src/staging.js`](@/src/staging.js) `highlightConsumption(typed, pool, trail?)` is the pure function consumed by [`src/components/GameScreen.vue`](@/src/components/GameScreen.vue) to decide which loose tiles and player-word rows render as "claimed." `trail` is an optional array of per-letter entries describing the *origin* of each character currently in the typed input (see the entry shape below); it lets the caller distinguish identity-bound clicks from raw keyboard input. When `trail` is omitted, empty, or contains only `kind:'typed'` entries (no identity-bearing entries), the function falls through to the legacy `findConsumption` + `partialHighlight` greedy-multiset path — so all pre-existing 2-arg call sites continue to work unchanged. When `trail` contains any `kind:'word'` or `kind:'loose'` entry with a valid `sourceIndex`, those indices are pre-claimed and a residual greedy pass runs only over the `kind:'typed'` letters against the *unclaimed* tiles/words. This hybrid keeps partial-cover hints for raw typing while preserving identity for click-staged tiles. Trail entries take one of three shapes:

  | `kind`    | fields                                  | meaning                                            |
  | --------- | --------------------------------------- | -------------------------------------------------- |
  | `'loose'` | `sourceIndex: int`, `letter: string`    | a specific loose-tile index was clicked            |
  | `'word'`  | `sourceIndex: int`, `letter: string`    | one letter of a clicked word row (one per letter)  |
  | `'typed'` | `letter: string`                        | a keyboard-typed letter; no identity binding       |

  Identity entries with an out-of-range `sourceIndex` (`< 0` or `>= pool.words.length` / `pool.looseLetters.length`) are silently ignored, which lets the caller carry stale entries across re-renders without crashing. Identity-claimed tiles are excluded from the greedy pass for `typed` letters — so a typed `r` after clicking tile 0 (R) picks tile 1, not tile 0 again.
- **Dictionary.** [`src/dictionary.js`](@/src/dictionary.js) `loadDictionary(json, lemmasJson?)` builds three indices: a `Set` for `isWord` lookups, a `signatureIndex: Map<sortedLetters, words[]>` for anagram subset detection, and a `lemmaIndex: Map<string, string[]>` populated from `lemmasJson.lemmas` (the precomputed `data/lemmas.json` payload). The `lemmaIndex` is part of the dictionary's public API and is consumed by [`src/anagramRules.js`](@/src/anagramRules.js) when invoking `isTrivialInflection`. The second argument is optional for back-compat in tests; when omitted, `lemmaIndex` is empty (so trivial-inflection checks pass through unless an override matches). Loaded once per session by `App.vue`.
- **Persistence layer.** [`src/storage.js`](@/src/storage.js) is a pure-function module over an injected storage shim. `browserStorage()` returns `window.localStorage` after a probe write (to detect Safari Private Mode where quota is 0), or a no-op shim. Records are stored under key `anagrams:v1` with shape `{ schemaVersion: 1, records: { 'YYYY-MM-DD': { score, longestWord, durationMs, completedAt, history? } } }`. `history` (an optional `[{word, parents}]` array of plays in submission order) is written only when `recordDailyResult` is called with a `history` field; old records that predate the field continue to load unchanged. `recordDailyResult` is **keep-first per date** (Wordle convention) — re-playing a date does not overwrite. `currentStreak` counts trailing consecutive local-calendar days and preserves the streak when today is missing but yesterday is present (so the streak is shown until the user actually breaks it). `bestScore`, `hasCompletedDate`, and `recentDays` are pure derivations over `records`. **Streak, best, and the recent-days view are never persisted as denormalized state — they are recomputed from `records` on each read.**
- **Recent-days window.** `recentDays(records, todayDate, count = 7)` returns an oldest-first array of `{ date, played, record? }` entries ending at `todayDate`. The `date` strings are local-calendar labels; it walks days backward via the same `shiftDate` helper used by `currentStreak` (opaque-string `±N`-day arithmetic on a `YYYY-MM-DD` label), so the window aligns with the streak/storage-key date convention. `record` (the full underlying entry — `score`, `longestWord`, `durationMs`, `completedAt`, and `history` when present) is attached only when `played` is true. The home-screen calendar in [`src/components/HomeScreen.vue`](@/src/components/HomeScreen.vue) reads these fields directly to populate its day-detail popover (score, longest word, formatted duration), and uses `history` to drive a per-day Share button via [`src/share.js`](@/src/share.js); no separate per-day persistence is required.
- **Share text contract.** [`src/share.js`](@/src/share.js) `generateShareText` produces a Wordle-style block: header `Anagrams <date|Random> — <score>`, an emoji grid where each played word becomes one row of `🟨` (carried letters from parents) followed by `🟩` (newly added letters), and a footer joining `Longest <N>`, `Time <m:ss>`, and (daily mode only, when `streak ≥ 2`) `Streak <N>` with ` · ` separators. Assembly is centralized in a private `assemble(header, rows, footer)` helper so the truncated and untruncated paths share one format definition.
- **Share text Twitter-budget truncation.** When the assembled text would exceed `MAX_WEIGHTED_LENGTH = 260` weighted units (a conservative budget under Twitter's 280-cap weighted model), `generateShareText` keeps only the first `HEAD_KEEP = 3` rows and the last `TAIL_KEEP = 3` rows of the emoji grid, with a single literal `… +N more` line spliced in between (where `N = rows.length − 6`). Header and footer are emitted verbatim regardless. Truncation is suppressed entirely when `rows.length ≤ HEAD_KEEP + TAIL_KEEP` — for ≤ 6 rows the elision would drop fewer than 1 row, so the function returns the un-truncated text even if it overshoots the budget. Weighted length is computed by a private `weightedLength` helper using the rule `cp <= 0x7F ? 1 : 2` (ASCII = 1 weight, everything else = 2), a deliberate over-approximation of the `twitter-text` v3 weighted model that keeps 🟨/🟩 (each = 2) accurate and tolerates a few weight units of slop on rare punctuation within the 260→280 safety margin.

### Things to Know

- **Date string is the user's local calendar day.** [`src/dateKey.js`](@/src/dateKey.js)'s `todayLocal()` (`formatLocalDate(new Date())`) returns `YYYY-MM-DD` from local-time accessors (`getFullYear`/`getMonth`/`getDate`), matching the NYT Wordle/Connections local-midnight reset convention. The same string is the PRNG seed *and* the storage record key, so they cannot drift. The seed/key is **locked at game start**: `App.vue` captures it once into `game.value.date` in `startGame`, so a session that crosses local midnight finishes on the date it began. `shiftDate` in [`src/storage.js`](@/src/storage.js) uses `Date.UTC` math purely as opaque-string `±N`-day arithmetic on the `YYYY-MM-DD` label — it is timezone-agnostic and not a semantic UTC commitment.
- **Storage failures are swallowed.** `loadStore` returns the empty store on missing-key, JSON parse failure, schema-version mismatch, and `getItem` throwing. `recordDailyResult` returns `{ wasNewRecord: false }` when `setItem` throws (quota exceeded). Callers must not assume a write succeeded.
- **`isNewBest` must be computed before recording.** [`src/components/App.vue`](@/src/components/App.vue) `endGameWithResult` snapshots `bestScore(loadStore(...).records)` *before* calling `recordDailyResult`, then compares the new score against that previous best. Reading after the write would always tie or lose.
- **Replays of a date never update stats.** Because `recordDailyResult` is keep-first, replaying an already-completed daily returns `wasNewRecord: false` and `isNewBest` stays `false` even if the replay scored higher.
- **Random mode never touches storage.** `endGameWithResult` only records when `game.mode === 'daily' && game.date`. Random mode's score screen will show `streak` (from the cached `stats`) but `isNewBest` will always be `false`.
- **Subset enumeration is exponential and capped.** Both `anagramRules.canFormWord` and `scoring.hasLoosePoolAnagram` enumerate subsets via bitmask; `canFormWord` short-circuits to `letters-unavailable` when `pool.words.length > 16`, and `hasLoosePoolAnagram` caps loose-letter subset size at `MAX_LOOSE_SUBSET = 8`.
- **Pool word `parents` are the share-text input.** [`src/share.js`](@/src/share.js)'s emoji grid relies on each history entry's `parents` array (set by `submitWord` from the consumed words). If `parents` were empty, every tile in that word would render as green (newly added).
- **Share-text truncation chose head+tail over re-encoding to preserve the per-row 🟨/🟩 split.** Long-form Wordle variants (e.g. 64ordle) typically re-encode rows into denser glyph sets when they overflow Twitter's limit. We deliberately do not: the per-row yellow-then-green pattern is the share's most informative signal — it encodes how much of each play was stolen vs newly added — and re-encoding loses that. Head + tail + count summary keeps the first 3 and last 3 rows byte-for-byte identical to their untruncated form, so the structural meaning of every shown row survives. The constants (`260`, `3`, `3`) are not exported; tests in [`tests/share.test.js`](@/tests/share.test.js) assert observable behaviour (overflow stays ≤ 280, elision text format, ordering) rather than the constants themselves. Common-case daily plays (typical 5–10 words) never trip the overflow branch and remain byte-identical to the prior output.
- **Schema versioning is strict-equal, but only breaking changes require a bump.** `loadStore` rejects any payload whose `schemaVersion` does not strict-equal the current `SCHEMA_VERSION` in [`src/storage.js`](@/src/storage.js), and treats it as empty (no migration). Removing or renaming a field — or changing the meaning of an existing one — therefore requires bumping the version. **Additive-optional fields (e.g. `history`) do not.** New writers attach the field; older records continue to load and consumers feature-detect it (`record?.history?.length`). No migration framework exists yet; the next genuinely breaking change will introduce one.
- **`data/lemmas.json` is committed and must stay in sync with `data/dictionary.json`.** `buildLemmaIndex` filters lemma candidates against vocabulary membership, so adding or removing words from the dictionary without rerunning `npm run build-lemmas` can leave inflection coverage stale (missed rejections, never spurious ones). The build-time pipeline produces the lemma list as the deduped union of `wink-lemmatizer`'s `verb`/`noun`/`adjective` outputs, dropping identity entries and any candidate not present in the vocabulary.
- **No runtime lemmatizer.** `wink-lemmatizer` is a `devDependencies` entry imported only by [`scripts/build-lemmas.js`](@/scripts/build-lemmas.js). The shipped JS bundle therefore does not include it; `data/lemmas.json` is fetched in parallel with `data/dictionary.json` on first game start.
- **`highlightConsumption` is back-compat with 2-arg callers.** The third `trail` parameter is optional. Callers that omit it (and tests that do) get the legacy greedy-multiset behaviour identical to the pre-trail implementation. The two paths are mutually exclusive: as soon as *any* identity entry appears in the trail the function commits to the trail-driven hybrid path; with only `kind:'typed'` entries it still takes the legacy path so behaviour matches a vanilla typed-string caller. This is the contract that lets the GameScreen component pass `trail.value` unconditionally without breaking the no-clicks-yet case.
- **Trail identity is the source of click-to-remove correctness.** The reason `highlightConsumption` accepts `trail` rather than re-deriving everything from `typed` is that a typed-string-only model loses information when the rack contains duplicate letters: the highlight set is "leftmost-of-letter per claim," so click-to-remove on the leftmost-claimed tile reverts the highlight to the rightmost identical-letter tile (visual impression of un-clicking the wrong one). Storing identity per click lets the caller splice the *clicked* entry directly. See [`src/components/docs.md`](@/src/components/docs.md) for the GameScreen-side click-handler decision tree.
