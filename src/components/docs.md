# Noridoc: components

Path: @/src/components

### Overview

- Vue 3 SFC layer for the Anagrams SPA. Three screen components (`HomeScreen`, `GameScreen`, `ScoreScreen`) plus the orchestrator (`App.vue`).
- Components are presentational — game state lives in domain modules under [`src/`](@/src), and persistence lives in [`src/storage.js`](@/src/storage.js). Components receive data via props and emit user intent via events.

### How it fits into the larger codebase

- [`src/components/App.vue`](@/src/components/App.vue) is the only file in the project that imports both [`src/storage.js`](@/src/storage.js) and the domain modules. It is the integration boundary between persistence, domain, and UI.
- Screen routing is a single `screen` ref (`'home' | 'game' | 'score'`) in `App.vue`. There is no router. Transitions between screens are the only points where stats are recomputed.
- `GameScreen` mutates the game by calling [`src/game.js`](@/src/game.js)'s pure functions (`drawTile`, `submitWord`) and re-binding the result locally; `App.vue` only learns the final result via the `end` event.
- `ScoreScreen` is the only consumer of [`src/share.js`](@/src/share.js). It assembles the `generateShareText` argument from its props (including the `streak` passed down from `App.vue`).

### Core Implementation

- **App.vue — orchestrator.** Owns `screen`, `game` (`shallowRef`), `finalResult`, `dict` (lazy-loaded), `stats` (`{ streak, best, completedToday, recent }`), and `isNewBest`. `computeStats()` reads storage via `loadStore(browserStorage())` and derives all four values, including `recent: recentDays(records, today, 7)` for the home-screen calendar strip. Stats are recomputed on transitions to `home` (`returnHome`) and on score-screen entry (`endGameWithResult`).
- **Share surfaces — dual buttons.** Both screens that surface share output (`ScoreScreen` and the `HomeScreen` day-detail popover) render a primary "Share" button (emoji grid via [`generateShareText`](@/src/share.js)) **and** a secondary "Copy alt text" button (narrative via [`generateShareAltText`](@/src/share.js)). Each button has its own copy-feedback state (`copied`/`copiedAlt` on `ScoreScreen`; `copiedDay`/`copiedAltDay` on `HomeScreen`). The two buttons are independent — clicking one does not affect the other's label. Both are gated identically on `result.history` (or `entry.record.history`) being a non-empty array.
- **App.vue — daily recording flow.** `endGameWithResult` only writes for `mode === 'daily'`: it snapshots the previous `bestScore` *before* calling `recordDailyResult`, then sets `isNewBest = wasNewRecord && result.score > previousBest`. This ordering is required because `recordDailyResult` is keep-first and returns `wasNewRecord: false` for replays. The call also forwards `history: result.history` so the persisted record carries the play history needed by the home-screen per-day Share button.
- **App.vue — date convention.** `todayLocal()` (imported from [`src/dateKey.js`](@/src/dateKey.js)) returns a `YYYY-MM-DD` string built from local-time accessors and is used both as the daily-mode `date` passed to `createGame` and as the storage record key, keeping the PRNG seed and storage key aligned. The puzzle resets at the user's local midnight (NYT Wordle/Connections convention).
- **HomeScreen.** Receives `loading`, `streak`, `best`, `completedToday`, `recent` props. Renders a stats row only when `streak > 0 || best > 0` (so first-time players see a clean home). When that condition holds *and* `recent.length > 0`, also renders a `.home-calendar` strip — a horizontal 7-cell window (oldest left, today right) with day-letter labels indexed off the UTC weekday. Cell rendering is split by `entry.played`: **played cells render as `<button type="button" class="home-calendar-cell played">`** (Tab-reachable, `aria-haspopup="dialog"`, `aria-expanded` bound to whether this cell is the open one, `aria-controls` pointing at the popover id `home-day-popover`); **unplayed cells stay as `<div role="img" class="home-calendar-cell">`** (non-interactive). The `.today` modifier is applied to the rightmost cell in either branch. Wrapped in `<div role="group" aria-label="Last 7 days activity">` with per-cell `aria-label` and `title` (e.g. "Today, score N, Friday April 25" / "Not played, Monday April 21"). Shows a `✓ Played today` sub-label on the Daily button when `completedToday` is true. Emits `start` with `'daily'` or `'random'`.
- **HomeScreen — day-detail popover.** Owns transient UI state for a single optional popover: `selectedIndex` (index into `recent`, or `null`), `panelPos` (`{top, left}` in viewport coords), `cellRefs` (array of trigger button DOM nodes, populated via `:ref="setCellRef(i)"`), `panelRef` (the panel DOM node), and `copiedDay` (transient share-button feedback). Clicking a played cell calls `onCellClick(i, ev)`, which computes the panel position from `ev.currentTarget.getBoundingClientRect()` (centered horizontally on the trigger, 8px below the trigger's bottom, clamped to `[8, viewportW - PANEL_W - 8]` so it never escapes the viewport with `PANEL_W = 220`), then sets `selectedIndex`. The panel is rendered via `<Teleport to="body">` as `<div role="dialog" aria-modal="false" tabindex="-1">` with `position: fixed` styling — teleporting to `body` keeps it out of any parent stacking context so the fixed coordinates are correct. The panel body is derived from `selectedEntry = recent[selectedIndex]` and shows the human date heading, `record.score`, `record.longestWord` (falling back to em-dash when empty), and `record.durationMs` formatted as `m:ss` via `formatDuration`. Includes a close `×` button (`aria-label="Close"`) at the top-right.
- **HomeScreen — popover Share + Copy-alt buttons.** Two `.action-btn` buttons are rendered inside the panel — a primary Share and a sibling "Copy alt text" — both gated on the same `canShareSelected` predicate (`Array.isArray(selectedEntry.record.history) && history.length > 0`). Records written before the optional `history` field existed (or with `history: []`) therefore show neither button; there is no degraded mode. The two handlers (`shareSelectedDay` / `copyAltSelectedDay`) share a `buildSelectedShareArgs(entry)` helper so the args object is constructed in one place; they differ only in which serializer they call ([`generateShareText`](@/src/share.js) vs [`generateShareAltText`](@/src/share.js)). Both `await navigator.clipboard.writeText(text)`; on resolve each sets its own copy-feedback flag (`copiedDay` / `copiedAltDay`) for 2s (button label flips to `Copied!`); on reject each falls back to `window.prompt('Copy your result:', text)`. **Past-day shares always pass `streak: 0`** for both serializers because streak is a "right now" concept defined relative to today's local-calendar date; retroactively reconstructing the streak as it stood on a past date is out of scope. As a result, the share footer for a past-day share never carries `Streak <N>` — only `Longest` and `Time`. **Truncation flows through automatically** for both forms: emoji shares go through Twitter-weighted-budget elision; alt-text shares go through row-count elision. Both use the same `HEAD_KEEP`/`TAIL_KEEP` constants documented in [`src/docs.md`](@/src/docs.md) without any popover-specific handling. **Both copy-feedback flags and their pending timers (`copiedDay`/`copiedTimer` and `copiedAltDay`/`copiedAltTimer`) are reset on every popover close** — close button, Escape, outside-mousedown, or the `recent` prop changing — and on cell-to-cell switch via `onCellClick`, so reopening or switching always starts in the un-copied state for both buttons. **Race guard:** `shareSelectedDay` and `copyAltSelectedDay` each capture `selectedIndex.value` as `startedFor` before awaiting `navigator.clipboard.writeText`; if the popover state has changed by the time the promise settles, both the success and rejection branches return early, so a delayed clipboard resolve cannot flash `Copied!` (or fire `window.prompt`) on a popover the user never shared from.
- **HomeScreen — popover dismiss & focus invariants.** A single `watch(selectedIndex, ...)` is the central state machine for open/close side effects. **On open** (`val !== null`): registers `mousedown` and `keydown` listeners on `document`, then `await nextTick()` and `panelRef.focus()`. **On close** (`val === null` with `prev !== null`): unregisters both document listeners and refocuses `cellRefs[prev]` so keyboard focus returns to the originating trigger. Dismiss paths: `closePopover()` is called by (a) the close button's `@click`, (b) `onDocKeydown` when `key === 'Escape'`, and (c) `onDocMousedown` when the event's `composedPath` includes neither `panelRef` nor any element in `cellRefs` (excluding triggers from the outside-click set lets a different trigger reopen via its own `@click` without a flicker). A separate `watch(() => props.recent, ...)` clears `selectedIndex` whenever the `recent` array changes (e.g. a new daily play swaps the strip). `onBeforeUnmount` defensively removes both document listeners.
- **GameScreen.** Holds a local mutable game ref, calls domain `submitWord` / `drawTile` to advance, and emits `end` with the `result` from `endGame` when the face-down pile is exhausted. Rack tiles and player-word rows are rendered as `<button type="button">` elements (with their existing `.tile` / `.word-row` classes preserved) so they are reachable by Tab and activate on Enter/Space via native button semantics. Staging routes through `@click`; a no-op `@mousedown.prevent` is kept on those buttons to suppress input blur on pointer click. Per-letter `<span class="tile">` decorations inside a word-row button are `aria-hidden="true"` so AT reads only the row's `aria-label`.
- **GameScreen — staging-trail model.** The component owns `trail = ref([])` (an ordered array of `{kind, letter, sourceIndex?}` entries — see [`src/docs.md`](@/src/docs.md) for the entry shape) and exposes `typed` as a *writable* `computed({get, set})` rather than a plain ref. The getter joins trail letters into the string used by `v-model` on the `<input>`. The setter compares the incoming string against the trail-derived previous string and chooses one of three branches: append-only (new ends with old) pushes new `kind:'typed'` entries for the suffix; backspace-only (new is a strict prefix of old) pops the matching number of tail entries (preserving identity entries when they sit before the cursor); any other divergence (paste, mid-string edit, programmatic `setValue`) **demotes** the entire trail to a fresh array of `kind:'typed'` entries — losing identity but staying consistent. The `consumption` computed passes `trail.value` as the third arg to [`highlightConsumption`](@/src/staging.js), so the highlight set is identity-driven where possible and falls back to greedy matching for typed residue.
- **GameScreen — click handler decision tree.** Both `onTileClick(i)` and `onWordClick(i)` follow the same three-step priority on each click:

  1. **Identity-bound entry exists** for this `sourceIndex` → splice it (tile click splices the single entry; word click splices *all* entries with `kind:'word' && sourceIndex === i` atomically). This is the "click-to-un-click" path that fixes the duplicate-letter leftmost regression.
  2. **No identity entry but the index is in `consumption` (greedy claim)** → the highlight came from `kind:'typed'` residue; remove the corresponding letter(s) from the *typed* entries via `removeFirstTypedLetter` (one call per word letter for a word click), preserving any other identity entries.
  3. **Otherwise (fresh click)** → push new identity entry/entries onto the trail (one for a tile click; N for a word click, one per letter).

  `onClear()` and the submit-success branch reset `trail.value = []` directly, **bypassing the `typed` setter**. Going through the setter would treat a clear as "any other divergence" and demote, which is identical in observable effect but semantically wrong (and the trail-direct write is cheaper).
- **ScoreScreen.** Receives `result`, `mode`, `date`, `streak`, `isNewBest`. Renders a streak/new-best row only when `mode === 'daily' && (streak > 0 || isNewBest)`. The `.score-actions` row contains a primary Share button (`onShare` → [`generateShareText`](@/src/share.js)) and, when `canCopyAlt` is true (`Array.isArray(result.history) && result.history.length > 0`), a sibling "Copy alt text" button (`onCopyAltText` → [`generateShareAltText`](@/src/share.js)). Both handlers route through a shared `buildShareArgs()` helper so the args object (`mode`, `date`, `score`, `longestWord`, `totalTimeMs`, `history`, `streak`) is constructed once. Each button has its own `copied` / `copiedAlt` ref that flips to `Copied!` for 2s on a successful clipboard write. Both fall back to `window.prompt` when `navigator.clipboard.writeText` rejects. The streak suffix (` · Streak <N>` for emoji form, `Streak <N>.` for alt form) appears only when `mode === 'daily' && streak ≥ 2`, enforced inside [`src/share.js`](@/src/share.js).

### Things to Know

- **`stats.streak` is the snapshot at score-screen entry, not pre-game.** When `endGameWithResult` runs, it calls `recordDailyResult` and then recomputes `stats`, so the streak shown on the score screen (and in the share text) reflects the just-finished play.
- **Random mode shows cached stats.** `ScoreScreen` is rendered with the same `stats.streak` regardless of mode, but the streak/new-best row is gated on `mode === 'daily'`. The share footer is gated identically inside `share.js`.
- **`isNewBest` resets on every transition.** `startGame`, `endGameWithResult`, and `returnHome` all reset `isNewBest` (the first two implicitly via the `bestScore` comparison, `returnHome` explicitly). It is only true on the score screen immediately after a daily play that beat the previous best.
- **Replays of a completed daily.** Because `recordDailyResult` is keep-first per date, a second daily play on the same local day returns `wasNewRecord: false`, `isNewBest` stays `false`, and `bestScore` reflects the *first* play's score even if the replay was higher.
- **Dictionary load is lazy and one-shot, and pulls two assets in parallel.** `ensureDictionary` issues `Promise.all` fetches for `data/dictionary.json` and `data/lemmas.json`, then `Promise.all`s the two `.json()` parses, then calls `loadDictionary(dictJson, lemmasJson)` to populate `dict.value` (including its `lemmaIndex`). Either fetch failing throws and surfaces via the `dictError` banner; the home buttons stay enabled but `startGame` will reject. There is no retry UI.
- **`shallowRef(game)` is intentional.** The game object is replaced wholesale by domain functions returning new objects; deep reactivity is unnecessary and would be wasteful given the immutable update pattern.
- **No event bus, no provide/inject.** All cross-screen state flows through `App.vue` props/events. Adding any new persisted value means updating `computeStats`, the `stats` ref shape, and the relevant screen's props.
- **HomeScreen popover: only-one-at-a-time, click-replaces.** `selectedIndex` is a single `ref<number|null>`, so at most one popover panel exists in the DOM at any time. Clicking a different played cell while one is open simply overwrites `selectedIndex` and recomputes `panelPos` against the new trigger — there is no explicit close-then-open dance. The `watch` callback receives `(val, prev)` with both non-null in this case and runs the open branch; focus is moved into the new panel and document listeners stay registered (the addEventListener call is idempotent under repeated registration, but in practice this branch only registers when transitioning from `null`).
- **HomeScreen popover: outside-mousedown excludes triggers.** `onDocMousedown` returns early when the event path includes any `cellRefs` entry. Without this, clicking a played cell would fire two handlers in order — the document `mousedown` (which would close), then the cell's `click` (which would reopen) — producing a flicker and breaking the "click a different cell to switch" test.
- **HomeScreen popover: panel is teleported to `body`.** The panel uses `position: fixed` and is rendered through `<Teleport to="body">` so it cannot be clipped by any ancestor's `overflow` or stacking context. Its z-index (`1000` in [`style.css`](@/style.css)) sits above the rest of the home screen.
- **HomeScreen popover: recent record fields are read defensively.** Every consumer of `entry.record` uses optional chaining (`entry.record?.score ?? 0`, `entry.record?.longestWord || '—'`, `formatDuration(entry.record?.durationMs)`), so an `entry.played === true` with a missing `record` will not throw. This matches the [`recentDays`](@/src/storage.js) contract that `record` is attached only when `played` is true and is the full underlying entry — but does not require any specific shape.
- **Staging source-of-truth is `trail`, not `typed`.** The `<input>` is bound to a writable `computed` named `typed` whose getter is `trail.value.map(e => e.letter).join('')`. The `<input>`'s `.value` is therefore a derived view, and the array of trail entries (each carrying optional `sourceIndex` identity) is the canonical model. The pattern mirrors chip / multi-token inputs in industry-standard libraries (Downshift `useMultipleSelection`, shadcn-vue `TagsInput`, PrimeVue `Chips`): the visible input is a transient editor for the next entry; the entry array is the model.
- **Keyboard-vs-click is unified through the trail.** Clicking a tile or word row mutates `trail` directly with identity-bound entries; typing into the input goes through the `typed` setter and produces `kind:'typed'` entries. Both handler families call `refocusInput()` so keyboard users can interleave typing and clicking without a manual Tab back. `onTileClick`, `onWordClick`, and `onClear` all refocus.
- **Click-to-remove preserves clicked-tile identity, not letter-count.** Before the trail model, clicking a duplicate-letter tile that was greedy-claimed un-highlighted the *leftmost-of-letter* tile (because the highlight was recomputed from typed-string multiset matching). The trail-as-source-of-truth model splices the specific clicked entry by `sourceIndex`, so the visual feedback matches the user's click target. This is the v4 known-limitation fix tracked in [`CURRENT-PROGRESS.md`](@/CURRENT-PROGRESS.md) "Sticky-claim or staged-array model."
- **Trail clears on submit success, not on draw.** A successful `submitWord` call rebuilds `pool.looseLetters` (consumed letters removed, others re-indexed), invalidating every `sourceIndex` in the trail — so submit-success resets `trail.value = []` (alongside `game.value = result.game`). A `drawTile` call only *appends* to `looseLetters`, leaving existing indices stable, so trail entries survive across draws. Invalid submissions also leave the trail intact (so the player can correct one letter without re-clicking everything).
- **`typed` setter demotes on divergence.** When the setter sees a string change that is neither an append-suffix nor a backspace-prefix (e.g. paste, mid-edit, `wrapper.setValue` in tests), it replaces the entire trail with a flat array of `kind:'typed'` entries matching the new string. This is a deliberate safety net: it preserves the typed-string contract for any external mutation path while losing identity, rather than trying to reconcile arbitrary string diffs against trail entries. After demotion, `highlightConsumption` falls through to the legacy greedy path because no identity entries remain.
- **Identity entries with stale `sourceIndex` are silently ignored.** [`highlightConsumption`](@/src/staging.js) bounds-checks each identity entry's `sourceIndex` against the current `pool.words.length` / `pool.looseLetters.length`. This means the trail can transiently carry stale entries during a re-render without crashing — but the GameScreen still proactively clears the trail on submit success because stale identity entries silently revert to no-highlight, which would visually look like un-clicked tiles.
- **Live-region policy in `GameScreen`.** `.feedback` is the only live region and carries `role="status"` with `aria-atomic="true"`. It is rendered unconditionally (empty text when no submission has happened) so the region exists in the DOM before its content changes — some screen readers miss announcements when the live region is created simultaneously with its text. `.score` is a static span on purpose: SR users get one announcement per submit (the feedback line) rather than two competing ones; the score is reachable on demand. `#timer` is explicitly `aria-live="off"` to prevent per-second tick spam. The typed input is paired with a visually-hidden `<label class="sr-only" for="word-input-field">` (the `.sr-only` clip-path utility lives in [`style.css`](@/style.css)), replacing placeholder-only labeling. The same stylesheet hosts the button-form reset, the `:focus-visible` outline that gives the button-shaped tiles, word-rows, action-btns, and mode-btns a visible keyboard focus indicator, and a trailing `@media (prefers-reduced-motion: reduce)` block that honours WCAG 2.3.3 by collapsing all animation/transition durations to `0.01ms !important` (preserving `animationend`/`transitionend` firing) via a `*, *::before, *::after` reset and additionally setting `transform: none` on `.tile.clickable:hover` so the universal duration reset does not still snap to the translated hover position. The `@keyframes pop-in` definition itself is left intact — users without the OS preference still see the rack tile pop-in.
